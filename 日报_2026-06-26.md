## 计划任务完成详情

### 过长拆分 (T-6.1)

当chunk超过max_chars(1200)或max_tokens(1200)时触发递归拆分。

```
split_oversized_chunks() 算法:

输入: 超大chunk (char_count > max_chars)
  1. 按优先级尝试分隔符:
     "\n\n" -> "\n" -> "。" -> "！" -> "？" -> "；"
     -> ". " -> "." -> "，" -> "、" -> ":" -> " " -> ""
  2. 在当前分隔符处切分为多个单元
  3. 递归合并相邻单元 (只要合并后不超过max_chars)
  4. 对仍超标的单元 -> 递归拆分
  5. 子chunk继承原始title_path和source_refs
  6. 相邻子chunk保留overlap_sentences句重叠

关键约束:
  - 只在分隔符位置切分, 不在句子中间截断
  - table/formula/code类型不参与递归拆分
```

验证: `test_recursive_split_handles_long_plain_text` — 输入~2000chars无换行文本, 配置max=180, 结果>=2 chunk, 全部<=180chars。

### 过短合并 (T-6.2)

三个函数协同处理短chunk:

```
merge_short_chunks()      — 主合并: 短chunk尝试与右侧相邻chunk合并
rebalance_short_chunks()  — 重平衡: 从长chunk尾部匀句子给短chunk
absorb_trailing_short_chunks() — 尾部吸收: 文档末端短chunk向前合并

合并条件 (can_merge_chunks):
  1. 右侧必须是normal类型 (table/formula/code不参与)
  2. 合并后 <= max_chars/max_tokens
  3. title_path相同 -> 可合并
  4. title_path有交集 + 任一 < min_chars -> 可合并
  5. 左chunk < 120 chars -> 可合并

重平衡 (P0修复核心):
  问题: 前块1076c > max, 后块269c < min -> 孤儿chunk
  解法: 从前块尾部逐句匀给后块, 直到recipient>=min_chars
        或donor已达min_chars下限
  效果: 中文命中率 50% -> 100%
```

验证: `test_merged_short_sections_keep_first_title_path` — 3个小节, min=300时两个短节不够各自300, 正确合并为一个chunk, title_path保持第一个标题。

### 目标长度区间控制 (T-6.3)

三级约束配置:

| 参数 | 字符 | Token | 说明 |
|------|------|-------|------|
| min | 300 | 100 | 低于触发合并 (token仅安全网) |
| target | 900 | 900 | 尽力逼近 |
| max | 1200 | 1200 | 超过触发拆分 |

双目标策略:
- 中文: 1char ~ 1.09token, target_chars(900)先触发 -> 字符主导
- 英文: 1char ~ 0.25token, token上限先触发 -> token主导
- Token计数: tiktoken cl100k_base 精确编码

## 技术难点与解决思路详解

### 难点1：中文多标题文档碎片化

**现象**：开题报告包含大量短标题（一至四级），原始分段逻辑在每个标题处强制收束，导致末段产生269字符的孤儿chunk（远低于min_chars=300），而前段已达1076字符无法再合并，命中率仅50%。

**根因分析**：标题边界优先策略与长度约束产生了冲突。标题是强边界信号，但如果当前chunk长度不足heading_flush_min_chars就遇到新标题，会跳过收束。问题出在最后一个小节：内容少、标题多，收束阈值（300字符）始终达不到，导致最后chunk累积了多个小节后突然超标。

**解决方案**：新增rebalance_short_chunks()函数，在合并阶段对所有chunk进行一次扫描，找到char_count低于min_chars的"孤儿chunk"，从其前一个"捐赠chunk"尾部逐句匀出内容，直到孤儿chunk达到min_chars或捐赠chunk已达下限。这种方式比简单合并更精细，不会破坏标题边界。

**效果**：中文文档命中率从50%提升至100%，全部14个chunk落在441~862字符范围内。

### 难点2：评估方法论本质缺陷

**现象**：用embedding余弦相似度评估标签质量时，规则模式得分（80%+）远高于LLM模式（34%）。但人工抽查发现LLM标签明显更准确。

**根因分析**：embedding余弦相似度衡量的是"措辞相似度"而非"语义相关性"。规则标签器通过TF-IDF直接从原文提取高频词作为标签，这些词和原文的embedding自然高度相似。LLM理解后用不同措辞表达同一概念（如"语义分段"→"结构感知切分"），虽然语义一致但向量距离变远。评估框架在反向奖励规则模式。

**解决方案**：引入LLM-as-judge作为主要评判手段。DeepSeek对每个标签判断"是否与原文主题语义相关"并给出置信度，这种评判方式基于语义理解而非字面匹配。同时保留jieba词级重叠作为无API时的回退方案（60%权重），embedding余弦降为辅助信号（40%权重）。

**效果**：标签评估从"15%（完全不可信）→65.3%（反映规则模式真实水平）→98.1%（LLM模式真实质量）"。

### 难点3：长表格语义保护与长度控制的矛盾

**现象**：DOCX文档中的长表格（50+行）被标记为特殊块（table类型），分段器对其整体保护不拆分。结果一个50行表格成为单一chunk，远超max_chars=1200。

**根因分析**：特殊块保护策略要求表格/公式/代码整体成块，这是为了保证语义完整性（表格行间有逻辑关联，不能随意拆散）。但保护策略没有考虑"超长表格"这种边界情况——当表格本身远超目标长度时，需要一个降级方案。

**解决方案**：在Document Loader层（而非Segmenter层）拆分长表格。Loader层有完整的行结构信息，可以精确控制拆分组。每组12个数据行+重复表头+分隔行，末组不足3行自动并入前组防止过小碎片。每个子表仍是独立的table类型DocumentBlock，分段器对其继续做特殊块保护。

**为什么不在Segmenter层拆分**：Segmenter层看到的是已经展平为文本的表格（Markdown格式），丢失了行结构信息，无法精确重复表头。在Loader层拆分可以利用python-docx的Table对象，准确识别表头行和数据行。
