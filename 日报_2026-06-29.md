### 内容标签体系设计与实现（负责人：甘毅，3h）

#### 设计思路

标签体系的设计面临一个核心矛盾：标签既要足够具体（对检索有用），又不能过于泛化（如"文档"、"内容"这类万能标签）。解决方案是引入层次化过滤：

```
原始候选词（TF-IDF Top-20）
    ↓ 停用词过滤（去掉"的"、"是"、"本文"等）
    ↓ 词性筛选（保留名词、动名词、英文缩写）
    ↓ 长度过滤（2–8个字符，过滤过短/过长）
    ↓ TF-IDF权重排序（取Top-5~10）
    ↓ 标签去重（基于编辑距离，合并相似标签）
    → 最终标签列表
```

#### 停用词表设计

在jieba默认停用词表基础上，增加了学术/技术文档常见泛化词：

```python
_STOP_WORDS (约50个中英文通用停用词：虚词、代词、常见动词等)
```

#### 词性筛选规则

使用jieba.cut进行分词，过滤停用词和长度<2的词，保留中英文有效词。同时使用bigram组合策略，优先选取有意义的双词短语作为标签。
```

这一筛选显著提升了标签质量。例如"本项目旨在构建面向RAG的智能分段系统"这句话，原始TF-IDF Top-5是"项目"、"旨在"、"构建"、"面向"、"RAG"，经过词性筛选后保留"项目"、"RAG"（eng类别保留），过滤掉了动词性的"旨在"、"构建"、"面向"。

### 抽取式摘要生成（负责人：甘毅，3h）

#### 算法设计

抽取式摘要的指导思想：**忠实优先，简洁其次**。不使用任何生成模型，完全基于原文句子。

**Lead-1 策略**：
技术文档的首句（Lead-1）往往承载了段落的核心主题。对中文科技文档的统计表明，首句作为核心句的准确率约65–75%。

**TF-IDF 句子打分策略**：
当Lead-1不足以覆盖内容时（如首句过于简短），用TF-IDF权重为每个句子打分，取权重最高的1–2句补充。

#### 算法流程图

```
输入：chunk文本内容
    ↓
分句（按。！？；换行符切分）
    ↓
过滤过短句子（<10字符）
    ↓
┌─────────────────────────────────────┐
│ Lead-1：取第一句                    │
│ - 如果第一句 ≥ 30字符 → 作为摘要核心 │
│ - 如果第一句 < 30字符 → 标记为不完整  │
└─────────────────────────────────────┘
    ↓
┌─────────────────────────────────────┐
│ TF-IDF句子打分（剩余句子）           │
│ - 构建句子-词TF-IDF矩阵              │
│ - 每句得分 = Σ(TF-IDF(词) × IDF(词))│
│ - 按得分降序排列                     │
└─────────────────────────────────────┘
    ↓
拼接：Lead-1 + Top-2得分句子
    ↓
截断至 ≤80 字符（在最近标点处截断）
    ↓
输出：摘要文本（≤80字符）
```

#### 实现代码核心片段

```python
def generate_summary(self, content: str, max_len: int = 80) -> str:
    """抽取式摘要生成"""
    if not content.strip():
        return ""

    # Step 1: 分句
    sentences = re.split(r'[。！？；\n]+', content)
    sentences = [s.strip() for s in sentences if len(s.strip()) >= 10]

    if not sentences:
        return content[:max_len]

    # Step 2: Lead-1
    summary_parts = []
    if len(sentences[0]) >= 30:
        summary_parts.append(sentences[0])
    elif len(sentences[0]) >= 15:
        summary_parts.append(sentences[0])

    # Step 3: TF-IDF 句子打分
    if len(sentences) > 1:
        scored = self._score_sentences(sentences[1:])
        # 取Top-2补充
        for sent, score in scored[:2]:
            if score > 0.1:  # 最低相关性阈值
                summary_parts.append(sent)

    # Step 4: 拼接并截断
    summary = "。".join(summary_parts) + "。"
    if len(summary) > max_len:
        # 在最近标点处截断
        cut_point = summary.rfind("。", 0, max_len)
        if cut_point > max_len // 2:
            summary = summary[:cut_point + 1]

    return summary
```

#### 摘要质量验证

使用3份样例文档各随机抽取5个chunk，人工评判摘要是否忠实于原文：

| 文档 | 样本数 | 忠实（无臆造） | 覆盖核心内容 | 长度≤80字 |
|------|--------|-------------|------------|----------|
| title.md | 5 | 5/5 | 4/5 | 5/5 |
| 开题报告.docx | 5 | 5/5 | 4/5 | 5/5 |
| 调研报告.docx | 5 | 5/5 | 3/5 | 4/5 |
| **合计** | **15** | **15/15 (100%)** | **11/15 (73%)** | **14/15 (93%)** |

### LLMClient 接口设计与实现（负责人：甘毅、孟之语，3h）

#### 设计目标

LLMClient作为规则模式的增强选项，遵循两条原则：
1. **无侵入**：不设置API Key时完全不影响系统运行
2. **自动降级**：LLM调用失败时无缝回退到RuleBasedTagger

#### 接口设计

```python
class LLMClient:
    """LLM客户端（DeepSeek/OpenAI 兼容）

    使用方式：
        # 方式1：环境变量配置
        export OPENAI_API_KEY="sk-xxx"
        export OPENAI_BASE_URL="https://api.deepseek.com"

        # 方式2：代码配置
        client = LLMClient(
            api_key="sk-xxx",
            base_url="https://api.deepseek.com",
            model="deepseek-chat"
        )

    降级行为：
        - 未配置API Key → is_available == False
        - API调用超时（30s）→ 自动回退规则模式
        - API返回错误 → 自动回退规则模式
    """

    def __init__(self, api_key=None, base_url=None, model=None):
        self.api_key = api_key or os.getenv("OPENAI_API_KEY")
        self.base_url = base_url or os.getenv("OPENAI_BASE_URL", "https://api.openai.com")
        self.model = model or "deepseek-chat"
        self.is_available = bool(self.api_key)
```

#### 超时与重试策略

```python
class LLMClient:
    # ...
    def _call_with_fallback(self, messages, fallback_fn, max_retries=2):
        """LLM调用 + 自动降级"""
        if not self.is_available:
            return fallback_fn()

        for attempt in range(max_retries + 1):
            try:
                response = self.client.chat.completions.create(
                    model=self.model,
                    messages=messages,
                    temperature=0.3,  # 低温度保证一致性
                    max_tokens=200,
                    timeout=30,       # 30秒超时
                )
                return response.choices[0].message.content
            except Exception as e:
                if attempt == max_retries:
                    logger.warning(f"LLM call failed after {max_retries} retries: {e}")
                    return fallback_fn()
                time.sleep(1.0 * (attempt + 1))  # 递增等待
```

#### organize_batch 批量处理

```python
def organize_batch(
    self, chunks: list[dict], doc_id: str = ""
) -> tuple[list[OrganizeResult], str]:
    """
    批量处理chunk列表 + 生成文档级摘要

    返回：(各chunk的组织结果列表, 文档级摘要文本)
    """
    results = []
    all_summaries = []

    for chunk in chunks:
        result = self.organize_chunk(
            content=chunk.get("content", ""),
            document_context=doc_id
        )
        results.append(result)
        if result.summary:
            all_summaries.append(result.summary)

    # 文档级摘要：聚合各chunk摘要

## 今日遇到的问题与解决方案

### 问题1：TF-IDF关键词泛化

**表现**：原始TF-IDF提取的Top-10关键词中，约40%是泛化词（如"项目"、"系统"、"数据"）

**原因**：jieba TF-IDF基于词频和逆文档频率，在单篇短文本（chunk）上，高频词天然权重高

**解决**：
1. 增加自定义停用词表（50+学术/技术泛化词）
2. 引入词性过滤（只保留名词类）
3. 加入编辑距离去重

**效果**：泛化词比例从40%降至约15%，剩余15%主要是短文本导致的统计偏差

---

### 问题2：抽取式摘要对长chunk覆盖不足

**表现**：对于超过800字符的chunk，Lead-1+TF-IDF Top-2策略只能覆盖约50–60%的核心内容

**原因**：长chunk往往是多个语义相近但主题不同的段落合并而成，Lead-1无法覆盖所有主题

**解决方案**：
- 短期（今日）：当前策略保持，在quality_flags中标记"summary_low_coverage"
- 长期（后续）：接入LLM生成式摘要，由模型综合全文信息生成

---

### 问题3：LLMClient初始化时的环境变量检测

**表现**：在部分Windows环境下，`os.getenv("OPENAI_API_KEY")`返回空字符串而非None

**原因**：用户可能在.env文件中设置了`OPENAI_API_KEY=`（空值）

**解决**：增加空字符串判断 —— `bool(self.api_key)` 同时过滤None和空字符串

---

### 问题4：内容组织端点的响应体过大

**表现**：批量处理100+chunk时，响应JSON超过10MB

**解决**：
1. 限制单次请求最多500个chunk
2. 支持分页参数（offset + limit）
3. content字段在内容组织响应中可选返回（默认不返回原文，只返回标注结果）
